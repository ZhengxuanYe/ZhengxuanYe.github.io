# npm

包管理工具

包管理工具是用于自动化软件包（库、框架等）的安装、更新、配置和删除的程序。它们极大地简化了依赖项管理和项目构建的过程。以下是几种流行的JavaScript/Node.js包管理工具的介绍：

npm (Node Package Manager)

  • 概述：npm 是 Node.js 的默认包管理器，随 Node.js 一起安装。它拥有世界上最大的软件注册表之一，包含数以百万计的开源包。
  • 功能：
    • 安装、更新和卸载包。
    • 管理项目的依赖关系。
    • 发布自己的包到 npm 注册表。
    • 运行脚本命令，如 npm start 或 npm test。
  • 使用方法：通过命令行工具与之交互，例如 npm install <package-name>。

Yarn

  • 概述：Yarn 是由 Facebook 开发并维护的一个快速、可靠且安全的依赖管理工具。它的设计目标是解决 npm 的一些性能问题，并提供更一致的依赖解析行为。
  • 特点：
    • 更快的安装速度，因为支持并行下载。
    • 使用锁文件 (yarn.lock) 来确保所有开发者在不同环境中获得相同的依赖版本。
    • 支持工作区，便于管理多包项目。
  • 使用方法：类似于 npm 的命令，如 yarn add <package-name>。

cnpm

  • 概述：cnpm 是淘宝提供的 npm 镜像服务，解决了国内用户访问官方 npm 注册表速度慢的问题。
  • 特性：
    • 提供更快的下载速度给中国大陆的开发者。
    • 同步官方 npm 注册表的数据，同时托管一些中国特有的包。
  • 使用方法：可以像使用 npm 一样使用 cnpm，只需替换命令前缀，如 cnpm install <package-name>。

npm init 是 Node.js 包管理工具 npm 提供的一个命令，用于初始化一个新的 Node.js 项目。它会引导用户通过一系列提示来创建一个 package.json 文件，该文件是项目的配置文件，定义了项目的元数据、依赖关系和其他设置。

使用 Node.js 模块解析算法

Node.js 会根据一定的规则来查找模块的位置。当你使用 require 时，它首先会在内置模块中查找，然后检查是否是核心模块。如果不是，它将继续按以下顺序搜索：

  1. 本地模块：按照提供的相对或绝对路径查找文件。
  2. node_modules 文件夹：如果路径指向的是 node_modules 文件夹内的包，则 Node.js 将尝试找到该包并加载其主模块。
  3. 父目录的 node_modules：如果在当前目录找不到，则继续向上一层目录查找，直到根目录为止。

环境依赖和生产依赖

在 Node.js 项目中，依赖项可以分为两类：开发依赖（Development Dependencies） 和 生产依赖（Production Dependencies）。

开发依赖（Development Dependencies）

开发依赖是指那些仅在开发环境中使用的工具或库。这些包通常用于构建、测试、文档生成等任务，在生产环境中并不需要它们。将这些依赖标记为开发依赖可以帮助减小生产环境中的安装体积，并确保只有必要的代码被部署到生产服务器上。

示例

  • Webpack 或 Babel：用于编译和打包 JavaScript 文件。
  • Mocha 或 Jest：用于运行单元测试。
  • ESLint：用于代码风格检查和静态分析。
  • Nodemon：在开发过程中自动重启 Node.js 应用程序。

生产依赖（Production Dependencies）

生产依赖是指应用程序在运行时真正需要的所有模块。这些依赖是必不可少的，因为它们直接参与了应用的功能实现。确保所有生产依赖都已正确安装对于保证应用程序能够正常工作至关重要。

示例

  • Express：Web 框架，提供路由、中间件等功能。

区分两种依赖的重要性

  1. 减少部署大小：通过明确区分开发和生产依赖，可以在部署时只安装必要的生产依赖，从而减小部署包的大小，节省磁盘空间和带宽。
  2. 提高安全性：某些开发工具可能包含潜在的安全风险，如果它们不在生产环境中存在，则可以降低攻击面。
  3. 优化性能：不必要的开发工具可能会增加启动时间和内存占用，特别是在容器化环境中。
  4. 简化 CI/CD 流程：持续集成/持续部署（CI/CD）流程可以针对不同的环境分别处理开发和生产依赖，确保每次构建都是干净且一致的。

管理依赖的最佳实践

  • 保持 package.json 清晰：确保 dependencies 和 devDependencies 部分清晰地区分开了各自的用途。
  • 定期更新依赖：经常检查并更新你的依赖，以获得最新的功能和安全修复。
  • 锁定依赖版本：使用 package-lock.json（npm）或 yarn.lock（Yarn）文件来固定依赖版本，确保所有开发者和生产环境使用相同的依赖组合。
  • 利用脚本命令：定义 npm 脚本来管理不同类型的依赖，例如 npm run build 可能只涉及开发依赖，而 npm start 则专注于生产依赖。

修改windows执行策略

在Windows操作系统中，执行策略（Execution Policy）是PowerShell的一项安全特性，它控制着可以运行的脚本类型。修改执行策略通常需要管理员权限。

要更改PowerShell的执行策略，请按照以下步骤操作：

  1. 以管理员身份打开PowerShell：

    • 按 Win + X 键，然后选择“Windows PowerShell (管理员)”或者“终端（管理员）”，这取决于你的Windows版本。
    • 或者你可以在开始菜单中搜索“PowerShell”，然后右键点击它并选择“以管理员身份运行”。

  2. 查看当前的执行策略：
    输入命令 Get-ExecutionPolicy 并按下回车键。这将显示当前设置的执行策略。

  3. 修改执行策略：
    输入命令 Set-ExecutionPolicy 然后加上你想要设置的策略名称。例如，如果你想要设置为RemoteSigned策略（允许本地脚本无限制运行，但远程脚本必须有数字签名），你可以输入：

  4. 确认更改：
    系统可能会提示你确认是否真的要进行更改。根据提示输入 Y 或 A 来确认。

以下是几个常见的执行策略选项：

  • Restricted: 默认设置，不允许任何脚本运行。
  • AllSigned: 要求所有脚本和配置文件都必须由受信任的发布者签署。
  • RemoteSigned: 只允许运行本地创建的脚本；从互联网下载的脚本必须有数字签名。
  • Unrestricted: 允许所有脚本运行，但会警告用户有关从互联网下载的脚本。
  • Bypass: 不会强制执行执行策略，适合用于开发环境。
  • Undefined: 清除当前作用域的执行策略。

请注意，在企业环境中修改执行策略可能受到组策略（Group Policy）的限制，所以如果是在工作或学校计算机上尝试更改此设置，可能需要联系系统管理员来帮助更改这些设置。此外，放宽执行策略可能会带来安全风险，因此请谨慎选择适当的策略级别。

PATH 环境变量

1. 基本概念

  • 定义：PATH 是一个操作系统环境变量，它是一个由分号（在 Windows 中）或冒号（在 Unix/Linux/macOS 中）分隔的目录路径列表。
  • 目的：当用户在命令行界面中输入一个可执行文件的名字时，操作系统会按照 PATH 列表中的顺序查找该文件，并尝试执行找到的第一个匹配项。

2. 查找机制

  • 过程：一旦用户输入命令并按下回车键，操作系统首先会在当前工作目录中搜索。如果找不到对应的可执行文件，它将依次检查 PATH 变量中列出的每个目录。
  • 效率：为了提高效率，系统可能会缓存某些信息，减少重复查找的时间开销。

3. 用户与系统级别的区别

  • 用户级：每个用户的 PATH 变量可以独立于其他用户而设置，允许个性化配置。
  • 系统级：影响所有用户的全局 PATH 设置。修改此级别需要管理员权限，并且会影响整个系统的命令解析。

4. 安全性考量

  • 风险：错误地配置 PATH 或者将其设置为不安全的位置可能导致安全漏洞，如命令注入攻击。攻击者可能利用这一点来执行恶意代码。
  • 最佳实践：只添加可信来源的路径到 PATH 中，并定期审查其内容以确保没有不必要的或潜在危险的条目。

5. 修改的影响范围

  • 临时修改：仅对当前命令行会话有效，关闭窗口后失效。适用于测试或短期需求。
  • 永久修改：对所有新启动的进程生效。对于已经运行的应用程序，则需要重启才能应用新的 PATH 设置。

6. 修改方法

通过图形界面
  1. Windows 系统属性对话框
    • 右键点击“此电脑”或“我的电脑”，选择“属性”。
    • 点击左侧的“高级系统设置”链接。
    • 在“系统属性”对话框中，点击“高级”选项卡下的“环境变量”按钮。
    • 在“环境变量”对话框中找到“系统变量”部分中的 Path 或者在“用户变量”中找到对应的 Path。
    • 选择 Path 并点击“编辑”。然后点击“新建”并输入新的路径。
    • 确认所有对话框以保存更改。
使用命令行工具

  1. CMD (仅限当前命令行会话)

    • 打开命令提示符窗口。

    • 使用 set 命令可以在当前命令提示符窗口中临时添加路径到 PATH。

  2. PowerShell (仅限当前会话)

    • 打开 PowerShell 窗口。

    • 使用 $env:PATH 变量可以为当前 PowerShell 会话添加路径。

  3. CMD (永久修改)

    • 打开命令提示符（以管理员身份运行）。

    • 使用 setx 命令来更新 PATH。

      注意：使用 setx 修改环境变量会影响所有新启动的进程，但不会影响正在运行的进程。

  4. PowerShell (永久修改)

    • 打开 PowerShell（以管理员身份运行）。

    • 使用 Set-ItemProperty 来修改用户级别的 PATH，或者使用 [System.Environment]::SetEnvironmentVariable 来修改系统级别的 PATH。

      这个命令会将新的路径追加到现有的机器（系统级别）PATH 中。对于用户级别的 PATH，将 "Machine" 替换为 "User"。

7. 实际应用案例

  • 开发环境配置：开发者经常需要添加特定版本的编译器、解释器或其他工具链所在的目录到 PATH 中，以便可以直接从任何地方调用它们。
  • 软件安装：许多应用程序安装时会自动将自己的二进制文件所在目录添加到 PATH 中，方便用户直接使用。

8. 注意事项

  • 冗余路径：避免在同一 PATH 中多次添加相同的路径，这不仅浪费资源还可能引起混淆。
  • 顺序问题：路径的排列顺序很重要，因为第一个匹配的可执行文件会被优先执行。如果存在同名但不同版本的程序，应根据实际需求调整顺序。
  • 备份原始设置：在进行任何重大更改之前，建议记录下原来的 PATH 设置，以便出现问题时可以迅速恢复。

npm安装包及其依赖

当使用 npm（Node Package Manager）安装一个包时，npm 会根据该包的 package.json 文件中的依赖声明来解析并安装所有直接和间接依赖。以下是关于 npm 安装包及其依赖的一些关键点：

1. dependencies 和 devDependencies

  • dependencies：这些是在生产环境中运行项目所必需的包。
  • devDependencies：这些是开发过程中需要的包，比如测试工具、构建工具等，它们通常不在生产环境中使用。

2. 依赖树

当你安装一个包时，npm 会递归地检查该包的所有依赖，并为每个依赖重复这个过程，直到构建出完整的依赖树。这棵树可以非常复杂，因为它不仅包含直接依赖，还包括那些依赖的依赖（即间接依赖或子依赖）。

3. package-lock.json

  • npm v5 及以上版本会在你安装包时自动生成或更新 package-lock.json 文件。这个文件锁定了项目的依赖关系图，确保所有开发者以及不同环境下的部署都能安装完全相同的依赖版本。
  • 这个文件有助于避免由于不同时间点安装的依赖版本差异而导致的问题。

4. node_modules 文件夹

  • 所有通过 npm 安装的包都会被放置在这个文件夹中。为了优化性能和磁盘空间，npm 会尽量扁平化依赖树，即如果两个包依赖于同一个包的不同版本，npm 会尝试在顶层安装该依赖的兼容版本。

npm start

npm start 是 npm（Node Package Manager）提供的一个非常常用且方便的命令，它用于启动应用程序。当你在项目的根目录下运行 npm start 时，npm 会查找 package.json 文件中的 scripts 字段，并执行与 "start" 关联的命令。

默认行为

如果 package.json 中没有定义 "start" 脚本，npm start 会默认尝试执行 node server.js。这意味着如果你的项目包含一个名为 server.js 的文件，而且这个文件是你的应用入口点，那么 npm start 将直接运行它。

  • 一致性：无论是在本地开发、持续集成环境中还是部署到生产服务器上，npm start 提供了一种一致的方式来启动应用程序。
